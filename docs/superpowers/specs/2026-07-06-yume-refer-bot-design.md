# Yume Telegram Refer Bot Design

Date: 2026-07-06

## 背景

Yume Bot 先继续走 Bots.Business + GitHub 模板路线，目标是尽快上线一个可试水的 Telegram 裂变邀请积分 Bot。后续如果用户量变大，再迁移到自有服务器版本。

当前通知频道为 `@yumeGptplus`，官方交流群为 `@yumeHubplus`。Bot 文案使用“小橘猫店员”风格，轻松、温和、带少量“喵~”，但不密集刷屏。

## 范围

本次完整版本包含：

- 活人验证随机算术题。
- 频道和交流群成员校验。
- 首次活人验证奖励、每日签到、邀请奖励。
- 邀请 2 小时观察期和手动刷新结算。
- 邀请明细和邀请排行。
- 卡密商城、自动扣积分发卡密。
- 管理员库存、订单、上下架、手动加扣积分命令。

本次不包含：

- 钱包、提现、链上提取。
- 自动频道公告。
- 自有服务器部署。
- 外部数据库或后台管理面板。

## 用户主流程

1. 用户发送 `/start` 进入欢迎页。
2. 如果没有通过活人验证，Bot 展示随机算术题入口。
3. 用户答对后设置 `human_verified = true`，首次通过奖励 `+1 积分`。
4. Bot 引导用户加入通知频道 `@yumeGptplus` 和交流群 `@yumeHubplus`。
5. 用户点击验证入群后，Bot 检查频道和群成员状态。
6. 通过后打开主菜单：
   - 我的积分
   - 每日签到
   - 邀请好友
   - 积分商城
   - 邀请排行
   - 使用教程
   - 我的卡密
   - 返回主菜单
7. 每日签到每 24 小时可领取 `+0.5 积分`。
8. 我的积分只显示当前总积分，不展示用户侧积分流水。

## 活人验证

活人验证使用随机算术题。用户答错 3 次后需要重新领取新题。未完成活人验证的用户不能进入完整菜单，也不能进入有效邀请待结算池。

完成活人验证本身奖励 `+1 积分`，频道和群验证不额外给积分。

## 邀请裂变

邀请奖励规则：

- 邀请人通过「邀请好友」获取专属链接。
- 新用户通过链接进入时，记录邀请关系。
- 被邀请人必须完成活人验证，并加入通知频道和交流群。
- 被邀请人通过频道和群校验后，进入 2 小时观察期。
- 满 2 小时后，如果被邀请人仍在频道和群里，邀请人点击「刷新邀请奖励」结算。
- 每个有效被邀请人给邀请人 `+1 积分`。
- 邀请奖励不设每日上限。

观察期规则：

- 未完成活人验证的用户不进入待结算。
- 满 2 小时后如果被邀请人退群，不结算，继续保留为待观察。
- 被邀请人之后重新加入频道和群，邀请人再次刷新时可以结算。
- 已结算用户不能重复结算。
- 刷新邀请奖励有 60 秒冷却。

通知规则：

- 被邀请人完成验证后，只通知邀请人前 5 次，避免刷屏。
- 结算成功时通知邀请人本次结算人数、获得积分、当前积分、剩余观察中人数。
- 没有可结算邀请时提示：

```text
小橘查过啦，暂时没有可结算邀请，喵~
可能是好友还没满 2 小时，或还没完成频道/群验证。
```

邀请页面展示：

- 专属邀请链接。
- 已绑定邀请数量。
- 观察中数量。
- 可结算数量。
- 已结算有效邀请数量。
- 当前积分。
- 按钮：刷新邀请奖励、邀请明细、邀请排行、复制邀请链接、返回主菜单。

邀请明细展示最近 10 个被邀请人昵称和状态：

- 未完成验证
- 观察中
- 可结算
- 已结算

邀请排行按已结算有效邀请人数排序，显示前 10 名和我的排名。

## 积分商城和卡密

商品为卡密类商品。用户兑换后自动扣积分并发 1 张卡密。

用户商城规则：

- 商城只显示有库存且已上架的商品。
- 售罄商品直接从商城隐藏。
- 商品按钮格式：

```text
立即兑换｜商品名｜价格积分｜剩余N份
```

- 用户点击商品按钮后不做二次确认。
- 兑换前检查用户仍在通知频道和交流群。
- 兑换前检查积分、商品上架状态和库存。
- 检查通过后扣积分、取 1 张卡密、保存订单、发送卡密给用户。
- 同一个商品可以重复兑换，只要积分和库存足够。
- 已兑换卡密永久保存在「我的卡密」。
- 每笔兑换通知管理员：用户、商品、扣除积分、剩余库存。
- 补货或上架后不自动发送频道公告，只给管理员回执。

兑换失败规则：

- 积分不足：不扣分，提示继续签到或邀请好友。
- 库存不足：不扣分，提示商品刚被换完。
- 用户退群：不扣分，提示先重新加入频道和交流群。
- 商品下架：不扣分，提示商品暂时收起来了。

## 管理员命令

只有 `ADMINS` 配置中的 Telegram 数字 ID 可以调用管理员命令。非管理员调用管理命令时提示无权限或静默拒绝。

### 批量导入卡密

命令格式：

```text
/addcards 商品名 价格
卡密1
卡密2
卡密3
```

规则：

- 同名商品再次导入时追加库存。
- 同名商品再次导入时用本次价格覆盖旧价格。
- 重复卡密自动跳过。
- 导入完成后返回成功数量、跳过重复数量、当前库存。
- 补货后不自动发频道公告。

### 库存查看

命令：

```text
/stock
```

展示每个商品：

- 商品名
- 当前价格
- 剩余库存
- 已卖数量
- 上架或下架状态

### 商品上下架

命令：

```text
/offproduct 商品名
/onproduct 商品名
```

规则：

- `/offproduct` 隐藏商品，但保留库存和已卖记录。
- `/onproduct` 恢复显示商品，继续使用原库存和价格。

### 订单查看

命令：

```text
/orders
```

展示最近 20 笔兑换记录：

- 用户
- 商品
- 扣除积分
- 完整卡密
- 时间

完整卡密只允许管理员查看。

### 手动加扣积分

命令：

```text
/addpoints 用户ID 数量
/cutpoints 用户ID 数量
```

规则：

- 管理员可手动给用户加积分或扣积分。
- 操作成功后通知用户积分变化和当前余额。
- 管理员收到操作回执。
- 扣积分不能把用户余额扣成负数；余额不足时拒绝操作。

## 数据设计

继续使用 Bots.Business 的 `Bot` / `User` property 和 `ResourcesLib`，不引入外部数据库。

用户级数据：

- 是否完成活人验证。
- 当前算术题答案和错误次数。
- 每日签到时间。
- 被谁邀请。
- 是否已经给邀请人结算。
- 用户已兑换卡密列表。

全局数据：

- 商品列表。
- 每个商品的库存、价格、已卖数量、上下架状态。
- 全局卡密去重集合。
- 最近订单列表。
- 邀请记录列表。
- 邀请排行榜统计。

积分余额继续使用 `ResourcesLib.userRes("balance")`。

## 模块设计

新增或调整的库：

- `libs/YumeConfig.js`：统一读取频道、群、积分名称、奖励数、观察期、冷却、管理员权限。
- `libs/YumeReferral.js`：记录邀请关系、计算邀请状态、执行 2 小时结算、生成排行和明细。
- `libs/YumeShop.js`：管理商品、库存、卡密去重、兑换、订单、上下架。

用户命令：

- `/start`
- `/human`

- `/settle_referrals`
- `/myreferrals`
- `/toplist`

管理员命令：

- `/addcards`

## 关键实现要求

- `/start` 不再直接给邀请人发邀请奖励。
- 被邀请人完成频道和群校验后，只进入待观察池。
- 邀请奖励只在邀请人点击「刷新邀请奖励」时结算。
- 所有用户可见文案保持中文小橘猫风格。
- 管理员命令必须统一做权限检查。
- 卡密兑换按“检查条件、扣分、取卡、写订单、发卡密、通知管理员”的顺序执行。
- 如果发货前检查失败，不扣分、不扣库存。
- 如果订单写入失败，不向用户发送卡密，并恢复本次扣除的积分和取出的库存；恢复失败时立即通知管理员人工处理。

## 验收标准

上线前至少验证这些流程：

- 新用户首次 `/start`，完成活人验证，获得 `+1 积分`。
- 未加入频道或群时不能进入完整主菜单。
- 每日签到 24 小时内不能重复领取。
- 邀请人不能通过自己的邀请链接绑定自己。
- 被邀请人未完成验证时不进入待结算。
- 被邀请人完成验证和入群后，未满 2 小时不能结算。
- 满 2 小时且仍在频道和群里，刷新后邀请人获得 `+1 积分`。
- 刷新邀请奖励 60 秒冷却生效。
- 邀请明细和排行榜展示正确。
- 管理员可以导入卡密，重复卡密被跳过。
- 商城只显示上架且有库存商品。
- 用户积分不足时兑换失败且不扣分。
- 用户退群后兑换失败且不扣分。
- 用户兑换成功后收到卡密，卡密永久出现在「我的卡密」。
- 管理员收到兑换通知。
- `/stock`、`/orders`、`/offproduct`、`/onproduct`、`/addpoints`、`/cutpoints` 权限和结果正确。

## 验证方式

本地验证：

```text
node --check commands/**/*.js
```

如果 PowerShell 通配不适配，逐文件或用 `Get-ChildItem` 执行语法检查。

导入验证：

1. 推送 GitHub。
2. 在 Bots.Business 重新 Git 导入。
3. 给 Bot 发送 `/setup`。
4. 用管理员号和测试小号完整走一遍活人验证、入群校验、邀请结算、卡密兑换。

## 后续迁移

如果后续迁移到自有服务器，需要迁移的数据包括：

- 用户余额。
- 邀请关系和结算状态。
- 商品库存。
- 已兑换卡密。
- 订单记录。

迁移不是本次工作范围。
